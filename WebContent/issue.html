*Issue : 
1. 중복된 아이디의 처리.
	1-1. 아이디는 같으나 부모가 다른 데이터의 처리 : 
		1-1-1.삽입 시 이슈 : 노드의 아이디에 접근 시 두개이상의 데이터가 변경 될 가능성 있기 때문에 부모의 아이를 포함 시켜야 함.
		1-1-2.미 삽입 시 이슈 : 의미가 있는 노드 일 경우에 신뢰성을 잃을 수 있음. 
	1-2. 아이디와 부모가 같은 데이터의 처리 :
		1-2-1.삽입 시 이슈 : 노드의 아이디에 접근 시 두개이상의 데이터가 변경 됨.
		1-2-2.미 삽입 시 이슈 : 의미가 있는 노드 일 경우에 신뢰성을 잃을 수 있음.
		
1). 해당 프로그램의 처리 : 미 삽입(Unique) 근거. 해당 프로그램의 id는 primary key의 속성을 띄어야 함.

2. 루트의 처리

2). 해당 프로그램의 처리 : 부모가 없는 노드들을 자식으로 함.

3. 부모가 없는 노드의 처리

3). 해당 프로그램의 처리 : 루트의 자식 노드로 간주. 즉 공백도 하나의 아이디로 써 사용.

4. 이미 트리인 데이터의 처리.
	4-1. 자식은 있지만, 현재 있는 노드와 id가 같은 서브트리의 처리 : 
		4-1-1.삽입 시 이슈 : 현재 데이터들이 X진 트리의 형태를 띄고 있을 때, 그 룰을 벗어 나게 될 수 있음.
		4-1-2.미 삽입 시 이슈 : 동일한 부모가 들어왔더라도, 자식의 데이터들이 의미가 있을 수 있음.  
	4-2. 부모의 아이디는 다르지만 자식의 id가 겹치는 트리 :
		4-2-1.삽입 시 이슈 : 노드의 아이디에 접근 시 두개이상의 데이터가 변경 됨.
		4-2-2.미 삽입 시 이슈 : 의미가 있는 노드 일 경우에 신뢰성을 잃을 수 있음.

4). 해당 프로그램의 처리 : 규칙 1)에 의해 중복 된 id는 추가 하지 않는 것으로 하며, 해당 프로그램은 X진 트리가 아니므로, id가 다르면 추가 하는 것을 원칙으로 함.

5. 부모 변경 처리
	5-1. 자식이 부모를 따라 가는 경우 : 해당 자식들이 부모보다 조부모에 연관이 있을 경우가 있음.
	5-2. 자식이 조부모를 따라 가는 경우 : 해당 자식들이 부모에 연관이 있는 경우가 더 많음.

5). 해당 프로그램의 처리 : 자식이 부모를 따라가는 것으로 함. 5-1, 5-2의 위험성으로 보나, 해당 문제를 해결하기 위한 이벤트 수로 보나, 부모 따라가는 것이 타당 함.

6. 부모 삭제 처리
	6-1. 자식이 부모와 같이 삭제 되는 경우 : 자식의 데이터가 필요 할 수 도 있음.
	6-2. 자식이 삭제가 안되는 경우 : 자식까지 필요 없어서 삭제를 하는 경우에 문제가 될 수 있음.

6). 해당 프로그램의 처리 : 규칙 5)에 의거 하여, 의미가 있는 자식이면 부모를 변경 한 뒤에 삭제 하면 되기 때문에 자식까지 삭제를 원칙으로 함.

7. 데이터 정렬
	7-1. unorder인 경우 : 상황에 따라 desc, asc를 지원 함으로 써 얻을 수 있는 이득이 있을 수 있음. 
	7-2. order인 경우 : 데이터 삽입 순서에 따라, 의미가 있을 수 있음.

7). 기본적으로 unorder로 하되 옵션으로 id를 기준으로 order를 할 수 있도록 함, order의 범위는 프로그램의 반응성을 위하여 해당 노드의 자식 노드들 만을 기준으로 함.
